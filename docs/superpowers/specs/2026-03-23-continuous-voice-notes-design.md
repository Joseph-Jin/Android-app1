# 连续语音笔记 App 设计文档

## 概述

这份文档定义了第一版泛用语音笔记 Android App 的产品设计。它面向“连续说、多次提交、逐条产出”的使用方式，而不是长文档编辑。

产品核心承诺是：

1. 按住说话。
2. 松手提交。
3. 前一条在后台处理中时，用户仍可继续录下一条。
4. 每次录音生成一张独立笔记卡片。
5. 用户查看清洗后的内容，必要时展开原始转写，并通过直接点击词语来修正 ASR 误识别。

这个产品不是文稿编辑器，而是一个连续采集语音片段、并把它们变成可单独复制笔记卡片的工具。

## 产品目标

让用户能够快速连续记录多条语音笔记，并把每条语音都转成一张“可信、易读、易复制、可轻量修正”的文本卡片。

## 目标用户

第一版面向泛用语音笔记用户，不先绑定医疗、法律或其他垂直行业。

## 核心原则

1. 连续采集优先。上一条在上传或处理中时，用户必须还能继续开始下一条录音。
2. 每次录音彼此独立。每一次按住到松手，都会生成一张独立卡片。
3. 修正围绕 ASR 误识别展开，不做泛化式改写。产品要帮助用户修正“语音识别听错了什么”。
4. 编辑必须轻量。用户应在原页面直接改词，而不是进入复杂编辑器。
5. 复制发生在整张卡片级别。第一版不支持多卡合并复制。

## 范围内

- Android App
- 云端音频上传、转写、清洗
- 本机历史记录
- 连续重复录音
- 每次录音生成一张独立卡片
- 清洗后文本与可展开的原始转写
- 每张完成卡片内部自动切句
- 词语或短语级别的点击替换
- 基于 ASR 误识别的推荐替换
- 手动输入替换
- 单卡复制
- 单卡失败重试

## 范围外

- iOS 或 Web 客户端
- 账号体系或跨设备同步
- 多卡批量合并复制
- 全文编辑器
- 录音过程中实时字幕
- 行业术语词库同步或共享云端词典
- 协作编辑
- 超出“返回处理结果所必需”范围的服务端持久化要求

## 用户体验

### 主页面

主页面同时承担录音入口和笔记流列表。

- 页面始终有一个明显的“按住录音”主控件。
- 用户松手后，立即在本地创建一张“处理中”卡片。
- 无论前面的卡片是否处理完成，页面都要立刻回到“可再次录音”的状态。
- 录音控件下方按时间顺序展示当前和历史卡片，这些历史记录保存在本机。

### 卡片状态

每次录音对应一张卡片，卡片只有以下三种状态：

1. 处理中
2. 已完成
3. 失败

#### 处理中卡片

- 在用户松手后立即出现
- 表示当前正在上传、转写、清洗
- 不会阻塞下一次录音

#### 已完成卡片

- 展示清洗后文本，并按句子切片陈列
- 提供单张卡片级别的复制操作
- 支持展开查看原始 ASR 转写
- 支持在句子陈列区域中直接点击词语进行修正

#### 失败卡片

- 仍保留在列表中
- 明确提示处理失败
- 提供重试入口
- 不阻塞继续录音

## 文本展示

### 清洗风格

第一版采用“轻到中度清洗”：

- 去掉口头禅、重复和明显停顿
- 略微提高可读性
- 保留原本表达顺序与核心意思
- 不主动重组为文章或成稿

### 切句展示

卡片完成后，清洗后的内容会被切成句子级片段用于展示。

- 句子直接展示在卡片内
- 交互入口不是“先选句子再编辑”
- 切句的目的主要是让长内容更易读，并为词语修正提供局部上下文

### 词语与短语交互

所有词语都可点击。

- 点击单位是完整词语或短语，不是单个字
- UI 可以额外强调“更像 ASR 误识别”的词语，帮助用户更快定位
- 这种高亮只是辅助，不限制点击范围；所有词语都必须可点

目标修正行为示例：

- “玩你” 可能需要修正成 “我爱你”
- “Web coding” 可能需要修正成 “Vibe Coding”

这些都属于 ASR 误听或误识别，而不是普通文案润色错误。系统在推荐时应优先考虑同音、近音、外来词误听、专有名词误识别等情况。

## 修正流程

当用户在已完成卡片中点击任意词语或短语时，系统需要：

1. 展示当前选中的词语或短语
2. 展示它所在的完整句子上下文
3. 展示基于 ASR 误识别的推荐替换
4. 支持用户一键点击推荐词直接替换
5. 支持用户手动输入任意替换内容

### 修正规则

- 替换后要立即更新当前卡片在本地显示的清洗文本
- 第一版修正只作用于当前卡片
- 不自动把替换结果同步到其他卡片
- 推荐替换优先解决 ASR 错误，不做风格化改写推荐

## 核心流程

1. 用户按住录音按钮
2. 用户说话
3. 用户松手
4. App 先在本地创建元数据并生成一张“处理中”卡片
5. App 把音频放入上传队列
6. 后端接收音频并创建处理任务
7. 后端执行 ASR 转写
8. 后端执行受限的轻中度清洗
9. 后端对清洗结果做切句，并生成面向 ASR 误识别修正的词语或短语推荐
10. 客户端收到结果后，把卡片更新为“已完成”
11. 用户复制卡片内容，或直接点词修正

## 系统架构

### Android 客户端职责

- 按住录音与松手结束
- 本地上传队列管理
- 立即创建处理中卡片
- 笔记流与本机历史展示
- 轮询或接收任务完成状态
- 句子渲染
- 词语或短语点击处理
- 本地替换结果应用
- 卡片级复制
- 失败卡片重试

### 后端职责

- 接收上传音频
- 创建处理任务
- 执行 ASR
- 执行受限清洗
- 对清洗文本切句
- 生成面向 ASR 误识别的词语或短语推荐
- 把结构化结果返回客户端

### 本地存储职责

设备本地需要保存：

- 卡片元数据与状态
- 用于失败重试的音频本地引用（如需要）
- 原始转写
- 清洗后文本
- 切句结果
- 替换推荐
- 用户已应用的本地替换结果
- 卡片时间戳

## 建议的数据结构

具体实现方式后续可以调整，但 implementation plan 至少要保留以下概念。

### Note Card

- `id`
- `createdAt`
- `status`（`processing` | `completed` | `failed`）
- `audioLocalPath` 或等价的本地音频引用
- `rawTranscript`
- `cleanText`
- `sentences[]`
- `copyText`
- `errorMessage`

### Sentence Slice

- `id`
- `text`
- `tokens[]`
- `rawSourceSpan` 或等价引用（如果有必要）

### Token / Phrase

- `id`
- `text`
- `startIndex`
- `endIndex`
- `isAsrSuspicious`
- `suggestions[]`

### Suggestion

- `value`
- `reason` 或可选的推荐来源说明

## API / 处理契约

implementation plan 需要定义一种后端接口契约，至少支持以下生命周期：

1. 上传单条录音音频
2. 返回服务端任务 ID 或笔记 ID
3. 查询或接收当前状态
4. 在完成时获取结构化处理结果

结构化结果至少应包含：

- 原始转写
- 清洗后文本
- 句子切片结果
- 用于直接点击交互的词语或短语切分结果
- 对应词语或短语的推荐候选（如有）

## 异常处理

### 网络失败

- 卡片进入失败态
- 用户可手动重试
- 不影响其他录音

### 后端处理较慢

- 卡片持续停留在处理中
- 用户仍可继续录下一条

### 推荐不准确

- 用户可以忽略推荐，直接手动输入替换内容

### ASR 误识别严重

- 用户可以展开原始转写做对照
- 用户仍然可以直接对可见词语或短语做手动修正

### 队列堆积

- 客户端本地需要保持提交顺序
- 队列变长不能锁死录音入口

## 2026-03-24 实现与验证记录

当前实现已经落到以下能力：

1. 用户可连续录多条语音，并生成多张独立卡片
2. 卡片支持 `处理中 / 已完成 / 失败` 三态
3. 已完成卡片支持按句展示与词语点击替换
4. 推荐替换和手动输入替换都能立即生效
5. 复制输出为当前修正后的整张卡片文本
6. ASR provider 与清洗 provider 已拆成独立配置入口

### 已完成的自动化验证

- 后端：
  - `cd backend && pytest app/tests -v`
  - 结果：`25 passed`
- Android：
  - `cd android && ./gradlew :app:testDebugUnitTest`
  - 结果：`BUILD SUCCESSFUL`
- 本地健康检查：
  - 启动 `uvicorn app.main:app --host 127.0.0.1 --port 8001`
  - 请求 `GET /health`
  - 结果：返回 `{"status":"ok"}`

### 手工联调状态

本轮没有完成完整的真机/模拟器手工联调，原因不是代码构建失败，而是当前环境的 `adb` 不稳定。

- `cd android && ./gradlew :app:installDebug`
- 结果：失败
- 直接原因：`could not install *smartsocket* listener: Address already in use`

因此，下列人工检查仍需要在稳定设备环境下补齐：

1. 连续录三条语音，确认产生三张卡片
2. 第一张处理中时，第二张和第三张仍可继续录
3. 一张卡片失败后可单独重试
4. 任意词语可点，推荐和手动替换都生效
5. 复制输出为当前修正后的整卡文本
6. provider 配置改动后，后端仍能在不改业务代码情况下切换模型实现

## 当前已知限制

1. 如果用户正在按住录音时发生旋转屏幕或 Activity 重建，这次录音会被取消
2. “所有词语都可点”的体验质量依赖后端切句和 token/phrase 粒度
3. provider 抽象已预留，但当前仓库仍只内置 `openai` 作为实现示例，接入更便宜模型仍需要补 provider 实现与注册

## 质量要求

### 产品质量

- 录音闭环应足够直接、顺手
- 用户必须容易理解“每松手一次就生成一条独立笔记”
- 复制动作要明显且快速
- 词语修正步骤要尽量短

### 处理质量

- 清洗后的文本必须尽量贴近用户原意
- 推荐替换需要明显偏向 ASR 修正，而不是一般性润色
- 切词与可点击单位必须保持为完整词语或短语，不能拆成单字

## 测试重点

implementation plan 与后续验证至少要覆盖：

1. 快速连续录音时，是否能稳定生成多张独立卡片
2. 一张卡片处理中时，是否还能继续录下一条
3. 失败卡片是否能稳定重试
4. 完成卡片的切句展示是否正确
5. 可点击单位是否始终为完整词语或短语
6. 点击任意词语后，是否都能打开替换面板
7. 选择推荐替换后，卡片内容是否立即更新
8. 手动替换后，卡片内容是否立即更新
9. 复制动作是否复制当前已修正后的卡片内容
10. 原始转写是否可按需展开对照

## 规划阶段待定项

以下问题应在 implementation plan 中决策，但不属于产品需求未定：

- Android 具体技术栈与 UI 框架
- 上传方式与任务轮询机制
- 后端具体技术栈
- 处理服务的最终结构化 schema
- ASR 疑似误识别高亮是否默认常显，还是可以做更弱的视觉提示

这些属于实现选择，不属于产品层面的悬而未决。

## 给 Implementation Plan 的约束建议

implementation plan 应把它视为一个边界清晰的单一产品：

- Android 端连续语音采集客户端
- 云端逐条音频处理链路
- 本地优先的卡片历史
- 词语级 ASR 修正交互

第一版 implementation plan 不应发散到账号、同步、流式 ASR 或多平台支持。
